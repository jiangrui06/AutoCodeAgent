## 一、现状盘点

| 维度 | 当前状态 | 缺口 |
| --- | --- | --- |
| **工作流** | Planner → Coder → Linter → Scanner → Executor → Fixer 闭环 | 缺少测试驱动、代码审查、项目级多文件生成 |
| **交互** | CLI + Gradio Web | Web 仅输出文本，缺少文件树、代码编辑器、Diff 对比 |
| **语言** | 只生成 Python | 无法处理 JS/TS/Java/C++/Go 等 |
| **上下文** | 单轮需求 + 长期记忆 | 缺少对已有代码库的理解（RAG） |
| **安全** | 子进程隔离 + AST 扫描 | 沙箱仍可访问本地文件，缺少完全隔离容器 |
| **部署** | 本地运行 | 没有 Docker/云端部署方案 |

---

## 二、目标定位

> **一个能读需求、读代码库、生成/修改多文件、自动测试、支持多语言的个人编程助手。**

---

## 三、分阶段规划

### 第一阶段：把现有体验做扎实（1–2 周）

**目标**：当前功能不翻车、Web 好用、新手零门槛。

| 编号 | 功能 | 说明 |
| --- | --- | --- |
| 1.1 | **Web 文件树** | 左侧显示 `auto_generated_code/` 目录，点击直接查看历史文件 |
| 1.2 | **代码高亮/预览** | Web 中用 Markdown 代码块 + 行号展示生成的代码 |
| 1.3 | **Diff 视图** | 修复轮次之间显示代码差异，方便看改了什么 |
| 1.4 | **一键打开文件** | 按钮调用系统默认编辑器打开生成的 `.py` 文件 |
| 1.5 | **更好的空状态引导** | Web 首次打开时给出示例需求卡片（如“写计算器”“写爬虫”） |
| 1.6 | **错误经验沉淀** | 把修复过的错误和解决方案存入长期记忆，下次遇到类似报错直接套用 |
| 1.7 | **配置校验增强** | 启动时检查 `.env`、API 余额、Python 环境，给出中文提示 |

---

### 第二阶段：支持多文件项目生成（2–3 周）

**目标**：不再只生成单个脚本，而是生成完整项目结构。

| 编号 | 功能 | 说明 |
| --- | --- | --- |
| 2.1 | **项目结构规划节点** | Planner 输出文件清单：`main.py`、`utils.py`、`config.py`、`tests/` 等 |
| 2.2 | **多文件生成器** | Coder 按文件清单逐个生成，保持接口一致 |
| 2.3 | **文件引用图** | 记录模块间 import 关系，方便后续修改 |
| 2.4 | **批量保存** | 按项目目录保存，而不是全部塞进 `auto_generated_code/` 根目录 |
| 2.5 | **项目模板** | 内置 Flask/FastAPI/CLI/爬虫等常用模板 |
| 2.6 | **README 自动生成** | 给生成的项目自动写使用说明 |

---

### 第三阶段：测试驱动与质量保障（2–3 周）

**目标**：生成能跑、能测、质量可控的代码。

| 编号 | 功能 | 说明 |
| --- | --- | --- |
| 3.1 | **测试生成节点** | Coder 先写 `pytest` 用例，再写实现代码 |
| 3.2 | **TDD 模式开关** | `.env` 中开启 `TDD_MODE=true` |
| 3.3 | **覆盖率检查** | 用 `pytest-cov` 检查测试覆盖率 |
| 3.4 | **类型检查** | 集成 `mypy`，拦截类型错误 |
| 3.5 | **代码格式化** | 集成 `black`/`ruff`，生成后直接格式化 |
| 3.6 | **修复质量评估** | Judge 节点不仅看是否报错，还看测试是否通过 |
| 3.7 | **重复修复熔断** | 同一错误反复出现 2 次就停止，避免死循环 |

---

### 第四阶段：读懂已有代码库（3–4 周）

**目标**：Agent 不再从零写，而是能改现有项目。

| 编号 | 功能 | 说明 |
| --- | --- | --- |
| 4.1 | **代码库扫描** | 读取用户指定的项目目录，解析文件结构和依赖 |
| 4.2 | **RAG 向量检索** | 用 `faiss`/`chroma` 把代码切分索引，需求来时检索相关代码 |
| 4.3 | **代码摘要生成** | 给每个模块生成一句话功能说明 |
| 4.4 | **需求转改懂任务** | 用户说“给登录页加个验证码”，Agent 定位到相关文件并修改 |
| 4.5 | **Git 集成** | 修改前自动 `git diff`，修改后生成 commit message |
| 4.6 | **影响面分析** | 改动某个函数前，先找出所有调用方 |

---

### 第五阶段：多语言支持（3–4 周）

**目标**：一个 Agent 能写多种语言。

| 编号 | 功能 | 说明 |
| --- | --- | --- |
| 5.1 | **语言检测** | 根据需求自动判断生成 Python/JS/TS/Java/C++/Go |
| 5.2 | **语言专属 Linter** | JS/TS 用 `eslint`，Java 用 `javac`，C++ 用 `g++` |
| 5.3 | **多语言沙箱** | 按语言选择执行器（Node、JDK、GCC 等） |
| 5.4 | **依赖管理扩展** | 支持 `package.json`、`pom.xml`、`Cargo.toml` 等 |
| 5.5 | **语言模板库** | 每种语言提供常用项目模板 |

---

### 第六阶段：安全与部署（2–3 周）

**目标**：能放心运行、能方便部署。

| 编号 | 功能 | 说明 |
| --- | --- | --- |
| 6.1 | **Docker 沙箱** | 用 Docker 容器执行生成代码，完全隔离本地环境 |
| 6.2 | **网络沙箱** | 默认禁止生成代码访问外网，需要时单独授权 |
| 6.3 | **文件系统白名单** | 只允许访问指定目录，防止误删系统文件 |
| 6.4 | **Docker Compose 一键部署** | `docker-compose up` 跑起 Web 服务 |
| 6.5 | **API 服务化** | 提供 `/run` `/status` `/files` 等 REST API |
| 6.6 | **并发控制** | 支持多用户排队执行，避免资源冲突 |

## 五、推荐优先级

针对个人使用、最快看到效果的推进顺序：

1. **第一阶段**（Web 体验 + 错误经验沉淀）—— 立刻提升日常使用感受
2. **第三阶段**（测试驱动 + 格式化）—— 生成的代码质量稳定
3. **第二阶段**（多文件项目）—— 从脚本升级到项目
4. **第四阶段**（读懂代码库）—— 真正变成编程助手

---

## 六、下一步行动

建议从 **第一阶段** 开始，因为改动可控、见效快。具体可从以下两项之一入手：

- **1.6 错误经验沉淀**：把每次修复的「错误信息 + 修复方案」写入 SQLite/Obsidian，Fixer 节点先查经验再调用 LLM
- **1.1 Web 文件树**：在 Gradio 侧边栏展示 `auto_generated_code/` 目录，支持点击查看历史代码
