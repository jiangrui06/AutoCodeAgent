# 多模型多 Agent Code Agent 项目规划书

### 1.1 项目定位

从零研发一套内部使用的 Code Agent，不依赖外部模型 API。系统统一接入公司自部署模型，根据任务类型、模型能力和运行负载自动选择模型，并由主 Agent 调度专业子 Agent，完成需求理解、任务规划、代码开发、测试、审查和集成。
### 1.2 MVP 目标

- 接入至少两个自部署模型，并通过统一的 OpenAI-compatible API 调用。
- 主 Agent 能把开发目标拆成带依赖关系的任务 DAG，并按需调度子 Agent。
- 完成“需求理解 → 规划 → 开发 → 测试 → 审查 → 集成 → 最终验证”闭环。
- 每次模型选择、Agent 决策、工具调用、代码变更和验证结果均可审计。
- 并行开发使用隔离工作区，失败重试、资源消耗和高风险操作受到限制。
- 通过真实仓库评测，验证智能路由相对固定单模型是否有质量或资源收益。

### 1.3 MVP 非目标

- 不做无人值守的生产发布或自动合并主分支。
- 不支持子 Agent 无限创建新 Agent。
- 不首发 IDE 插件、跨项目长期记忆和学习型路由器。
- 不把“模型回答看起来不错”作为完成标准，必须用测试、静态检查和 diff 审查验证。

## 2. 模型策略

### 2.1 候选模型

| 逻辑角色 | 候选模型 | 初始用途 |
|---|---|---|
| `dense-planner` | `Qwen/Qwen3.6-27B` | 需求理解、仓库分析、任务拆分、复杂诊断和结果整合 |
| `moe-coder` | `Qwen/Qwen3.6-35B-A3B` | 边界明确的代码实现、测试编写和问题修复 |
| `reviewer` | 首版可映射到上述任一模型 | 独立代码审查和验收判断 |

说明：`Qwen3.6-31B-A3B` 不是本方案采用的准确型号，官方开源型号为 `Qwen3.6-35B-A3B`，总参数 35B、激活参数 3B。

具体模型只写入模型注册配置，业务代码依赖逻辑角色和能力标签，不写死模型名称。

### 2.2 路由原则

“Dense 负责规划、MoE 负责编码”只是首轮假设，不作为未经验证的固定结论。Phase 0 应在同一任务集上比较两类模型，再确定角色映射。

首版路由采用“规则过滤 + 能力评分 + 运行状态”的可解释方案：

1. 过滤不健康、无权限、上下文不足或不满足数据策略的模型。
2. 根据任务类型匹配候选模型。
3. 综合离线质量、当前负载、延迟和资源消耗评分。
4. 生成主模型、回退模型和预算限制。
5. 保存完整 `RoutingDecision`，供审计和离线评测。

路由器暂不使用另一个大模型做黑盒决策，避免首版复杂度和不可解释性过高。

### 2.3 初始路由规则

- 需求理解、跨模块分析、任务拆分：优先 `dense-planner`。
- 边界明确、输入文件有限的编码和测试任务：优先 `moe-coder`。
- 连续失败两次、影响范围扩大或涉及安全关键代码：回到规划/审查模型重新诊断。
- 模型过载或不可用时按配置回退，不允许静默切换到违反数据策略的模型。
- Review Agent 不复用 Implementation Agent 的结论，只读取需求、diff 和验证证据。

## 3. Agent 体系

首版保留 6 类核心角色，架构设计和仓库分析作为 Planner Agent 的能力，避免角色过细导致调度和上下文成本过高。

| Agent | 主要职责 | 关键输出 |
|---|---|---|
| Main Orchestrator | 维护目标、任务 DAG、预算、调度和终止条件 | 运行决策、最终状态与报告 |
| Planner Agent | 理解需求、分析仓库、设计方案、拆分任务和定义验收标准 | Requirement Spec、Repo Map、Task DAG |
| Coding Agent | 在指定文件和范围内修改代码并补充测试 | Patch、Change Report |
| Test Agent | 执行验证、定位失败并生成修复建议 | Verification Report |
| Review Agent | 独立检查正确性、安全性、可维护性和需求覆盖 | Review Verdict |
| Integration Agent | 按依赖集成补丁、处理冲突并执行最终检查 | Integrated Patch |

约束：

- 主 Agent 根据任务需要创建子 Agent，不要求每次运行全部角色。
- 每个子 Agent 只处理一个边界清晰的任务，不得自行扩大范围。
- 子 Agent 默认不能继续创建 Agent；层级、并发、重试和运行时间均有限制。
- 主 Agent 只能依据补丁、测试结果和审查结论判定完成，不能只相信文本声明。

## 4. 工作流程与系统架构

### 4.1 典型流程

```text
用户提交开发目标
        ↓
Planner Agent 理解需求、分析仓库并定义验收标准
        ↓
Main Orchestrator 校验任务 DAG 和风险边界
        ↓
Model Router 为每个任务选择模型
        ↓
Coding Agent 在隔离工作区开发并运行局部测试
        ↓
Test Agent + Review Agent 独立验证
        ↓
Integration Agent 集成通过的补丁并运行全量检查
        ↓
通过：输出代码、证据和报告
失败：生成有限次数的修复任务或转人工处理
```

### 4.2 核心架构

```text
CLI（MVP） / REST API（后续）
             |
             v
     Multi-Agent Orchestrator
       |       |        |
       |       |        +-- Policy / Budget / Approval
       |       +----------- Task DAG Scheduler
       +------------------- Agent Runtime
             |
             v
      Intelligent Model Router
       |                    |
       v                    v
 Model Registry       Evaluation Metrics
       |
       +-- dense-planner -> 自部署推理服务
       +-- moe-coder     -> 自部署推理服务
       +-- reviewer      -> 自部署推理服务

Agent Runtime -> Tool & Sandbox Gateway
                     |
          Files / Search / Command / Git / Test
                     |
              Isolated Workspaces

Orchestrator -> PostgreSQL / Artifact Store / Audit Log
```

### 4.3 技术基线

- 编排服务：Python 3.12、FastAPI、Pydantic、asyncio。
- 推理服务：生产候选优先评估 vLLM 或 SGLang；Ollama 仅作为个人开发或快速 PoC 选项。
- 持久化：PostgreSQL 保存任务、事件和审计元数据；大日志与补丁保存到产物存储。
- 工作区：每个写任务使用独立 Git worktree，并在容器或现有安全沙箱中执行命令。
- 可观测性：结构化日志为 MVP 必需；OpenTelemetry 和 Prometheus 在试运行阶段补齐。
- 测试：pytest、临时 Git 仓库、Fake Model Server 和 Fake Tool Executor。

## 5. 核心契约与状态

优先冻结以下版本化契约：

- `ModelProfile`：模型角色、端点、能力、上下文、并发、健康和数据策略。
- `RoutingRequest / RoutingDecision`：任务特征、候选过滤、评分、选择和回退链。
- `AgentRun`：目标、输入产物、允许工具、预算、状态和输出。
- `TaskNode / TaskDAG`：依赖、优先级、资源锁、重试和验收标准。
- `Artifact`：需求、Repo Map、补丁、测试和审查结果的版本化引用。
- `ToolCall / ToolResult`：参数、工作区、退出状态、输出摘要和证据位置。
- `ReviewVerdict`：通过/驳回、问题级别、证据和修复建议。

MVP 只实现会话内状态恢复和项目级 Repo Map，不引入向量数据库、跨项目长期记忆或自动知识学习。这些能力在核心闭环稳定后再评估。

## 6. 调度与安全规则

1. 任务依赖全部完成后才可进入 `READY`。
2. 同一文件或模块的写任务默认互斥；只读分析任务可以并行。
3. 并行写任务必须在独立 worktree 中执行，不能共享未提交状态。
4. 工具采用允许列表；网络默认关闭，按任务授权开放。
5. 禁止读取 `.env`、密钥目录和授权仓库之外的文件。
6. 命令必须有超时、输出上限和进程树终止机制。
7. 破坏性 Git 操作、工作区外写入、依赖安装和外部请求需要策略允许或人工审批。
8. 测试或审查失败时创建新的修复节点，最多重试 2 次，之后转人工处理。
9. 每个任务限制模型调用次数、Token、运行时间、并发和沙箱资源。
10. 所有模型决策、工具调用、代码变更和验证证据写入审计事件流。

## 7. 分阶段实施计划

### Phase 0：模型与方案可行性（2–3 周）

| 任务 | 关键产出 | 验收标准 |
|---|---|---|
| 冻结 MVP 边界与 ADR | 目标用户、接口、沙箱、审批和评测范围 | P0 问题有明确结论和依据 |
| 部署两类候选模型 | 统一 OpenAI-compatible API | 工具调用、结构化输出、取消和超时可用 |
| 建立能力与容量基线 | Dense/MoE 对照报告 | 获得质量、显存、吞吐、P95 延迟和并发上限 |

**Checkpoint A：** 只有模型能力、硬件容量和初始分工得到真实验证后，才进入编排开发。

### Phase 1：单 Agent 代码闭环（3 周）

| 任务 | 关键产出 | 验收标准 |
|---|---|---|
| 建立工程骨架和核心契约 | API、router、runtime、tools、storage、tests | lint、typecheck、unit 和 integration 通过 |
| 实现模型注册与可解释路由 | 模型健康、能力、评分和回退 | 路由可复现、可人工覆盖 |
| 实现单 Agent Runtime | 受控工具、结构化产物、预算和取消 | 单 Agent 能修改样例仓库并用测试验证 |
| 实现安全工作区 | worktree、沙箱、工具允许列表 | 越权路径和高风险命令被拒绝 |

**Checkpoint B：** 一个 Agent 能在隔离环境中完成真实小型代码任务，并留下完整审计证据。

### Phase 2：多 Agent 协作闭环（3–4 周）

| 任务 | 关键产出 | 验收标准 |
|---|---|---|
| 实现 Task DAG 调度器 | 依赖、并发、资源锁和有限重试 | 不执行非法 DAG；冲突写任务被串行化 |
| 落地 6 类 Agent | 角色模板和交接契约 | 主 Agent 能按需调度子 Agent |
| 实现补丁集成与回滚 | 版本检查、冲突任务和恢复机制 | 不覆盖用户改动；失败可恢复 |
| 实现测试、审查和修复循环 | Verification Report、Review Verdict | 失败可生成修复任务并安全停止 |

**Checkpoint C：** 在样例仓库完成“需求 → 规划 → 并行开发 → 测试 → 审查 → 集成”闭环。

### Phase 3：评测与内部试运行（2–3 周）

| 任务 | 关键产出 | 验收标准 |
|---|---|---|
| 建立端到端评测集 | 固定 Dense、固定 MoE、智能路由三组对照 | 智能路由至少在质量、时延或 GPU 消耗之一有稳定收益 |
| 补齐监控与运维 | trace、指标、告警和故障手册 | 单任务可还原完整执行链路 |
| 内部试运行 | CLI、试点报告、回滚方案 | 无未关闭的 P0/P1 安全问题，失败模式可控 |

**Checkpoint D：** 通过内部试点后再决定是否开发 REST API、IDE 插件和长期记忆。

## 8. 排期与资源

按 3–4 名工程师、GPU 与沙箱资源及时到位估算：

| 阶段 | 周期 |
|---|---:|
| Phase 0：可行性验证 | 2–3 周 |
| Phase 1：单 Agent 闭环 | 3 周 |
| Phase 2：多 Agent 闭环 | 3–4 周 |
| Phase 3：评测与试运行 | 2–3 周 |

**MVP 总周期建议按 10–13 周管理。** 若推理平台、GPU 集群或安全沙箱尚未就绪，应额外增加基础设施准备时间。

建议核心投入：

- 编排/后端工程师 2 名。
- 模型部署与评测工程师 1 名。
- 安全或运维支持 0.5–1 名。
- 产品/项目负责人可由技术负责人兼任。

硬件数量不能在缺少 GPU 型号、量化精度、目标上下文和并发数据时直接确定。Phase 0 必须先做容量压测，再冻结部署拓扑和预算。

## 9. MVP 验收标准

- [ ] 两个自部署模型均可通过统一 API 稳定调用。
- [ ] 在真实仓库完成至少 10 个覆盖规划、编码、修复和测试的评测任务。
- [ ] 主 Agent 能生成合法 DAG，并按需调度至少两个专业子 Agent。
- [ ] 并行写任务使用隔离工作区，不覆盖用户代码或彼此修改。
- [ ] 模型选择包含原因、评分、回退链和策略版本。
- [ ] 最终结果包含 diff、测试命令、退出状态和独立审查结论。
- [ ] 修复重试超过上限后明确停止，不产生无限循环。
- [ ] 智能路由相对固定单模型在成功率、P95 时延或 GPU 时间至少一项有稳定收益，且其他关键指标不显著恶化。
- [ ] 越权访问、高风险命令、提示注入和敏感信息泄漏测试通过。

## 10. 主要风险

| 风险 | 应对措施 |
|---|---|
| Dense/MoE 分工没有收益 | Phase 0 做同任务对照，不达标则调整或简化为单模型方案 |
| 模型显存或吞吐不满足并发 | 降低上下文、量化、限流或调整部署拓扑，并重新压测 |
| 多 Agent 增加成本却不提升质量 | 只对可独立、可验证的任务启用多 Agent，记录单 Agent 基线 |
| 主 Agent 拆分错误 | 冻结验收标准、校验 DAG、设置阶段 Checkpoint 和独立 Review |
| Agent 覆盖代码或泄漏数据 | worktree、沙箱、最小权限、网络默认关闭和审计日志 |
| 输出格式不稳定 | 强类型 schema、有限修复、失败显式化和人工接管 |
| 范围持续膨胀 | 将 API 产品化、IDE、长期记忆和学习型路由放到 MVP 之后 |

## 11. P0 决策清单

1. [ ] 确认 `Qwen/Qwen3.6-27B` 与 `Qwen/Qwen3.6-35B-A3B` 的权重、许可证和量化方案。
2. [ ] 确认 GPU 型号、数量、显存、网络拓扑、目标上下文和并发。
3. [ ] 在 vLLM 与 SGLang 中选择首个生产候选；Ollama 是否只用于本地 PoC。
4. [ ] 确认首发代码语言、仓库规模、构建系统和单任务最长时间。
5. [ ] 确认容器或现有安全沙箱方案，以及网络默认策略。
6. [ ] 确认自动执行、人工审批和禁止执行的动作清单。
7. [ ] 确认首轮评测任务、指标阈值和固定单模型基线。

## 12. 下一步行动

接下来 3–5 个工作日只做三件事：

1. 完成 P0 决策，冻结模型、硬件、推理平台、沙箱和评测范围。
2. 在真实 GPU 环境部署两个候选模型，验证工具调用、结构化输出、上下文、吞吐和稳定性。
3. 准备至少 10 个真实或脱敏仓库任务，建立 Dense、MoE 和智能路由的统一对照基线。

在 Phase 0 通过前，不启动复杂多 Agent 编排、长期记忆或产品化界面开发。
